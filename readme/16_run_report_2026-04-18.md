## TL;DR

Three runs were needed to get to green:

1. **Run 1** — uncovered two latent bugs in the steps renderer and
   one config bug (`OLLAMA_NUM_PREDICT=512` truncates feature plans
   for pages with >3 scenarios). Login completed; stewie's feature
   plan failed mid-LLM-call with truncated JSON.
2. **Run 2** — verified the steps-renderer fixes by re-rendering
   login from cache. Stewie still failed because the user's `.env`
   override was still 512.
3. **Run 3** — added `OLLAMA_NUM_PREDICT=2048` as a shell override.
   Both URLs reached `complete`. Stewie's feature plan returned 472
   valid-JSON tokens in 188 s.

Total artifacts produced: **2 POMs · 2 Gherkin features · 2 step
modules · 4 cached LLM plans · 2 extraction snapshots · 1 registry**.
All generated Python parses cleanly with unique function names. No
secrets were logged at any point during the run.

Defects surfaced:

- **Steps renderer dedupe bug** — the LLM emitted the same step text
  ("I click terms of service") under both `When` and `And/Given`,
  and the renderer emitted **two Python functions with the same
  name**. Visible in the first generated `tests/steps/test_login.py`.
- **Steps renderer arg bug** — `fill_email()` was emitted as a

  Would TypeError at runtime.

  plans for pages with multiple scenarios. The model's response was
  cut off mid-string and could not be parsed.
- **Single URL crash aborted the whole run** — the orchestrator did
  not catch `OllamaError` per-URL, so stewie's failure killed
  processing for any URLs after it.

## Bugs found and fixed

| # | Bug | Where | Fix | Verified by |
|---|-----|-------|-----|-------------|
| 1 | Resolver had no fallback when `.env` had `BASE_URL`/`LOGIN_URL` but no `AUTOCODER_URLS`. CLI exited with "no URLs". | `src/autocoder/intake/sources.py` | Added 4th-priority `settings_fallback` parameter to `resolve_urls`; CLI passes `[LOGIN_URL, BASE_URL]`. | 4 new unit tests in `tests/unit/test_url_sources.py`; live run logged `urls_source source=settings count=2`. |
| 2 | `OLLAMA_NUM_PREDICT=512` truncated stewie's feature-plan JSON mid-string → `OllamaError`. | `src/autocoder/config.py`, `.env.example` | Bumped default to **2048** (Phi-4's 16K context handles it easily). Updated `.env.example` with an explanatory comment. | Run 3 returned 472 well-formed tokens; `feature_plan_validated scenarios=3`. |
| 3 | Steps renderer deduped by `(keyword, text)` so the same text under two keywords produced two functions with the same Python name. | `src/autocoder/generate/steps.py` | Dedupe by **text only**; stack `@given/@when/@then` decorators on a single function. Added `_resolve_keywords` so `And`/`But` inherit the prior keyword. | 7 new unit tests in `tests/unit/test_steps_renderer.py`; `ast.parse` confirms `unique_names=True` for every generated file. |
| 4 | Steps renderer emitted `pom.fill_email()` with no args even when the method required `value: str` → TypeError at runtime. | `src/autocoder/generate/steps.py` | Cross-reference `POMPlan.methods[*].args`; if supplied < required, emit `raise NotImplementedError("...expects: value")`. | Visible in both `test_login.py` and `test_stewie.py`. |
| 5 | A single URL's exception (e.g. `OllamaError`) aborted the whole multi-URL run. | `src/autocoder/orchestrator.py` | Wrapped `_process_url(...)` in try/except inside `run_generate`; failing URL is marked `failed`, registry is saved, loop continues with the next URL. | Code in place; not exercised in run 3 because all URLs succeeded. |

## Limitations and observations

1. **Auth-first did not fire**, even though `LOGIN_URL` was set and a
   `LOGIN_USERNAME` was configured. Cause: the user's app uses
   Microsoft SSO; the credential page is on Microsoft's domain, so
   the app's own `/login` page has no `<input type="password">`. The
   classifier's "looks like login" heuristic
   (`autocoder/intake/classifier.py:_looks_like_login_page`) requires
   a password field to be present and therefore returns `False`.
   Both URLs were classified `public` and no `auth_setup` test was
   rendered. The system extracted the unauthenticated SSO landing
   page for both URLs.
2. **Both URLs share extraction fingerprint** `6cd2356ec6bed892`.
   That is consistent with point 1 — anonymous probes of `/login`
   and `/stewie` show the same SSO landing page. The orchestrator
   correctly produced separate POM/feature/step files keyed by
   slug, but their POM-plan inputs were near-identical, so the LLM
   was paid twice for nearly-identical work. Cross-fixture
   deduplication would be the cleanest fix; for now, listing only
   one URL when both yield the same content avoids the duplicate
   spend.
3. **Several generated steps raise `NotImplementedError`.** This is
   intentional. When the LLM emits a step that cannot be bound to a
   real POM method (e.g. "I should be logged in successfully"), the
   renderer emits an explicit `raise NotImplementedError("Implement
   step: ...")` so the test cannot silently pass. Authors must
   replace those bodies before the suite is meaningful.
4. **Selector quality on this app.** One element on each page is
   captured via XPath fallback (`html_1_body_1_div_1_div_1_div_2_...`)
   because it has no test-id, no accessible name, no label, and no
   placeholder. The runtime self-heal layer still handles it, but
   that selector is fragile. Adding `data-testid` attributes to the
   email input would let the inspector pick a stable selector.
5. **Some monitor / bash status reports were unreliable** during the
   run — the harness reported "completed exit code 0" while the
   underlying autocoder Python process was still mid-LLM-call. The
   source of truth was always `manifest/runs.log`, the registry, and
   `tasklist`. Future runs should rely on those three signals, not
   on bash exit codes from background invocations.

## Recommendations for the next run

1. **Update `.env`**: change `OLLAMA_NUM_PREDICT=512` to
   `OLLAMA_NUM_PREDICT=2048`. Without this, future feature-plan
   calls for pages with several scenarios will truncate again and
   raise `OllamaError`. The shell override used in run 3 was a
   temporary workaround; the `.env` value still wins on a normal
   invocation.
2. **Drop one of the two URLs** unless the app behaves differently
   when authenticated. Right now both produce the same anonymous
   extraction → wasted LLM budget. Either:
   - run an `autocoder generate` with only `LOGIN_URL`, then add
     `BASE_URL` once you have a session captured, or
   - improve the classifier to detect Microsoft SSO entry pages
     (e.g. by looking for the literal "Sign in with Microsoft"
     button) and treat them as `LOGIN`-kind.
3. **Add `data-testid` attributes to the email input** (and any
   other framework-styled fields). This will replace the XPath
   fallback with a stable `test_id` selector and remove a real
   future flake source.
4. **Implement the `NotImplementedError` step bodies** in
   `tests/steps/test_login.py` and `tests/steps/test_stewie.py`
   before running pytest, otherwise those scenarios will fail at
   the first non-bound step.
5. **Consider a renderer-version bump** to force re-rendering of
   already-`complete` URLs when `generate/*.py` changes. Today the
   only way to apply a renderer fix to old URLs is to manually
   reset their status (as done for login during run 2) or to use
   `--force` (which also re-spends LLM tokens).

## Audit: secrets exposure check

A grep over `manifest/runs.log` across all three runs:

```bash
jq -r '. | tostring' manifest/runs.log \
  | grep -E '(password|secret|token)=[^ ]' \
  | grep -v 'present=' | grep -v 'env='
```

returned **0 matches**. No credential values, bearer tokens, session
cookies, or `.env` raw lines were written to either the console or
`runs.log`. URL fields were sanitised with `logger.safe_url(...)`
(query strings + fragments stripped). This is the behaviour required
by `readme/15_logging.md` and the system held to it.
